# GetOffer 本地 Demo：Agent 可解释执行过程开发文档

## 1. 目标

在本地 demo 聊天链路中展示稳定、可控、面向用户的 Agent 执行阶段，而不是展示模型原始思维链。

统一展示 4 个固定阶段：

```text
已理解需求
正在检索/调用工具
已读取工具结果
正在生成建议
```

适用范围：

```text
普通聊天
主页快捷提示语
文件上传分析
一站式求职方案
简历生成
模拟面试 answer_text 分析
```

## 2. 明确不做

- 不展示百炼返回的原始 `thoughts` 文本。
- 不展示模型真实思考过程。
- 不把执行过程持久化到数据库。
- 不改变 MCP 工具名、Agent intent、OSS 路径。
- 不改变现有最终回答的数据库保存策略：刷新后只保留用户问题和最终回答。

## 3. 用户体验

用户发送消息后，在对话区域显示一个轻量过程面板。

建议 UI：

```text
Agent 正在处理
✓ 已理解需求
⟳ 正在检索/调用工具
○ 已读取工具结果
○ 正在生成建议
```

阶段规则：

- 已完成：显示对勾。
- 当前阶段：显示转圈动画。
- 未开始：显示灰色圆点。
- 最终答案开始输出后，过程面板保留在本轮回答上方或折叠为“执行过程已完成”。

推荐默认保留完整 4 行，不需要展开/折叠，便于 demo 观众理解链路。

## 4. 后端方案

### 4.1 新增流式事件

沿用现有：

```text
POST /api/agent/chat/stream
```

增强 SSE 事件类型：

```text
stage
delta
done
error
```

事件示例：

```text
event: stage
data: {"stage":"understood","label":"已理解需求"}

event: stage
data: {"stage":"tooling","label":"正在检索/调用工具"}

event: stage
data: {"stage":"observed","label":"已读取工具结果"}

event: stage
data: {"stage":"answering","label":"正在生成建议"}

event: delta
data: {"text":"这里是最终回答增量"}

event: done
data: {"conversationId":"conv_xxx","sessionId":"xxx"}
```

### 4.2 阶段触发逻辑

后端不直接暴露原始 thoughts，只把百炼返回情况映射为固定阶段。

基础阶段：

1. 收到合法请求并写入用户消息：

```text
已理解需求
```

2. 开始请求百炼 Agent：

```text
正在检索/调用工具
```

3. 满足以下任一条件：

- 百炼 SSE chunk 中出现 `output.thoughts`
- `thoughts[].action_type` 不为空
- `thoughts[].observation` 不为空
- 百炼开始返回最终答案前

映射为：

```text
已读取工具结果
```

4. 收到 `output.text` 第一段最终回答前：

```text
正在生成建议
```

### 4.3 百炼请求参数

在 `dashscopeProvider.ts` 的流式调用中增加：

```json
{
  "parameters": {
    "incremental_output": true,
    "has_thoughts": true
  }
}
```

说明：

- `has_thoughts=true` 只用于后端判断阶段，不原样转发给前端。
- 如果某个应用或模型不返回 thoughts，后端仍按固定时间点补齐 4 个阶段。

### 4.4 普通 chat 与文件上传

建议让前端聊天统一使用流式接口。

对于文件上传，目前已有：

```text
POST /api/agent/chat-with-file
```

后续有两种选择：

方案 A，推荐：

新增：

```text
POST /api/agent/chat-with-file/stream
```

使用 `multipart/form-data` + SSE 不方便直接组合，建议改为两步：

1. 文件上传到后端临时解析，返回 `fileContextId`。
2. 前端用普通 JSON 调 `/api/agent/chat/stream`，携带 `fileContextId`。

方案 B，较小改动：

文件上传链路继续非流式，但前端先播放固定 4 阶段，再收到结果后逐字输出。

本地 demo 推荐先采用方案 B，改动小、风险低。

## 5. 前端方案

### 5.1 数据结构

在 `ChatPage.tsx` 增加当前轮次执行阶段状态：

```ts
type AgentStageKey = "understood" | "tooling" | "observed" | "answering";

type AgentStage = {
  key: AgentStageKey;
  label: string;
  status: "pending" | "active" | "done";
};
```

固定阶段：

```ts
const agentStages = [
  { key: "understood", label: "已理解需求" },
  { key: "tooling", label: "正在检索/调用工具" },
  { key: "observed", label: "已读取工具结果" },
  { key: "answering", label: "正在生成建议" }
];
```

### 5.2 UI 组件

新增组件：

```text
client/src/components/AgentProcessPanel.tsx
```

职责：

- 接收阶段状态数组。
- 渲染对勾、转圈、灰点。
- 不渲染任何模型原始 thoughts。

### 5.3 ChatPage 改造

当前 `ChatPage` 使用：

```ts
agentApi.chat()
agentApi.chatWithFile()
```

后续改为：

- 普通聊天：优先使用 `agentApi.chatStream()`。
- 文件上传：先采用前端固定阶段模拟，再调用 `chatWithFile()`。
- 所有返回答案仍通过现有逐字输出逻辑展示。

### 5.4 Agent API 改造

新增：

```text
client/src/services/agentApi.ts
```

方法：

```ts
chatStream(payload, handlers)
```

handlers：

```ts
{
  onStage(stage) {}
  onDelta(text) {}
  onDone(data) {}
  onError(error) {}
}
```

## 6. 数据库策略

不新增表。

不修改 `messages` 表。

保存内容仍然只有：

```text
用户消息
最终助手回答
```

刷新页面后不恢复执行过程，这是预期行为。

## 7. 错误处理

如果百炼流式请求失败：

1. 当前过程面板停止转圈。
2. 显示助手消息：

```text
正在处理，请稍后重试。
```

3. 不保存失败的助手消息，或按现有错误兜底保存普通错误文本，二选一。

推荐：沿用现有前端错误展示，不保存失败助手消息。

## 9. 风险

- 百炼不同模型或应用可能不稳定返回 `thoughts`，所以前端展示必须依赖固定阶段，而不是依赖原始 thoughts 文本。
- 文件上传链路若要真正流式，需要额外设计两步上传，不建议本轮直接做。
- 若后续要展示 MCP 工具名，应先确认是否允许展示工具调用名称；当前方案只展示通用文案。

## 10. 推荐实施顺序

1. 新增 `AgentProcessPanel`。
2. 改造后端 `/api/agent/chat/stream`，输出固定 `stage` 事件。
3. 改造前端 `agentApi.chatStream()`。
4. 改造 `ChatPage` 普通聊天使用流式。
5. 文件上传链路先用固定阶段模拟。
6. 跑前后端 build 和本地链路验收。
